## Update Workflow

For every screen task:

1. Fetch the latest screen list or relevant screen source from the Figma Make file above.
2. Confirm the target screen still exists and whether route/screen scope changed.
3. Implement or update the screen.
4. Update this file:
   - `status`
   - `target files`
   - `notes`
   - `last updated`
5. If the screen change affects product/context docs, also run `context-sync-on-change`.

---

## Status Legend

- `not_started`: No meaningful implementation exists in the React Native app.
- `partial`: Some related UI exists, but it does not yet match the current Figma source.
- `in_progress`: Active implementation work is underway.
- `implemented`: Screen exists and is aligned closely enough to current app expectations.
- `blocked`: Implementation is waiting on dependency, API, or clarification.
- `n_a`: Not applicable to the current React Native app scope.

- Last synced from Figma Make routes: `2026-05-17`
- Source used: `src/app/routes.tsx`, `src/app/screens/Root.tsx`, `src/app/screens/Home.tsx`, `src/app/screens/NewEntry.tsx`, `src/app/screens/EntryDetail.tsx`, `src/app/screens/Streaks.tsx`, `src/app/screens/Onboarding.tsx`, `src/app/screens/Auth.tsx`, `src/app/screens/CreateAccount.tsx`, `src/app/screens/VerifyEmail.tsx`, `src/app/screens/SignIn.tsx`, `src/app/screens/Paywall.tsx`, `src/app/screens/SetupProfile.tsx`, `src/app/screens/Profile.tsx`, `src/app/screens/Settings.tsx`, `src/app/screens/Privacy.tsx`, `src/app/screens/Subscription.tsx`

Current implementation note:

- the mobile app now uses a React Navigation native-stack shell with root auth/paywall/legal routes and a nested authenticated app stack, so screen rows below describe the routed surfaces rather than the old stage-swapped full-screen host
- the app explicitly enables `react-native-screens` at startup and is pinned to `react-native-screens@4.25.0-beta.1` so the native-stack matches the current React Native 0.82 Fabric runtime
- the authenticated app stack now uses a subtle fade transition between screens, and the in-app legal browser modal uses a bottom-up modal presentation
- pushed screens inside the authenticated app stack now keep iOS swipe-back gestures, while the bottom-nav tab replacements remain fade-based shell switches
- RevenueCat purchase and restore failures across hosted and in-app paywall surfaces use shared app-owned copy so Test Store simulated failures do not expose raw SDK/store error text to users
- Restore purchases now stays on the current paywall screen and shows the shared `No purchases found` dialog unless RevenueCat returns the exact active `Journal.IO Pro` entitlement; backend plan attribution comes from that entitlement's product identifier rather than the selected card
- Completed purchases across hosted and in-app paywalls now land on the shared payment success screen; when entitlement sync is still catching up, the same success surface explains that premium access is updating instead of skipping ahead or showing a raw completion alert
- Hosted RevenueCat paywalls now ignore native dismiss callbacks while a purchase or restore is in progress, so checkout cannot route away before the shared payment success screen renders
- Closing the hosted post-auth main paywall now continues directly to the normal post-auth destination; the obsolete gift-wheel and custom discount routes, screens, and store state have been removed
- Purchasable prices now come only from exact RevenueCat package `priceString` values, preserving the user's App Store storefront currency rather than device locale or backend monetary defaults
